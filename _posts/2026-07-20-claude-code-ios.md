## 🆕 公式アップデート

- [Claude Code v2.1.215（2026-07-19）](https://code.claude.com/docs/en/changelog) — 主要変更点は 3 つ。**① `/verify` と `/code-review` スキルの自動実行廃止**：従来は Claude がコード実装後に自律的に `/verify` や `/code-review` を呼び出すことがあったが、v2.1.215 以降は明示的に呼び出さない限り実行されなくなった。iOS 開発ワークフローでの影響は Tips セクション参照。**② バックグラウンドセッションのゾンビデーモン修正**：`←`（バックアロー）や `/background` コマンドでバックグラウンドに退避したセッションがアイドル状態のままデーモンとワーカープロセスを永久に存続させ続けていた問題を修正。XcodeBuildMCP の長時間ビルドをバックグラウンドエージェントに任せて別作業をしているときに余分なプロセスが溜まる症状が解消。**③ 完了済みバックグラウンドセッションの削除不能バグ修正**：バックグラウンドサービスがアイドルに移行した後、`claude rm` やエージェントビューから完了済みセッションを削除しようとしても操作が効かなかったバグを修正。`claude agents` のセッション一覧が整理できるようになった。

## 🛠 GitHub の動き

- [Issue #79198 — Remote Control: iOS アプリで「PR マージ後の自動アーカイブ」設定 OFF にもかかわらずセッションが自動アーカイブされる（OPEN / 2026-07-19）](https://github.com/anthropics/claude-code/issues/79198) — Claude Desktop（Mac）と Claude iOS アプリで同一の Remote Control セッションを管理しているとき、「Auto-archive after PR merge or close」設定をオフにしているにも関わらず iOS アプリ側でのみセッションが自動アーカイブされてしまう問題。Mac のデスクトップアプリではセッションが「Connected」のまま残るが、iOS アプリ側では同じセッションがアーカイブ済みとして消えてしまう。報告者の仮説として「iOS アプリが Pull-to-refresh 時に独自のステイルセッションクリーンアップを走らせており、アカウント全体の設定とは独立して動作している」可能性が挙げられている。XcodeBuildMCP のビルドセッションを Mac に立てておき iPhone から Remote Control で状況を確認しに行ったら「セッションが消えている」という事態に繋がる。現時点では暫定回避策として Desktop でセッションをアーカイブしてから再開する方法しかない。

- [Issue #79175 — バックグラウンド Bash タスクが stdin を読もうとすると永久停止（フォアグラウンドでは問題なし）（OPEN / 2026-07-19）](https://github.com/anthropics/claude-code/issues/79175) — `claude agents`（バックグラウンドセッション）内から起動した Bash タスクが stdin からの入力待ちになると、処理を全く進めずに永久に停止してしまうバグ。フォアグラウンドセッションでは同じコマンドが正常に動く。XcodeBuildMCP + v2.1.212 の「MCP 自動バックグラウンド化」を組み合わせている場合：ビルドスクリプト中でパスワードや確認入力を求めるサブコマンドが呼ばれると Bash タスクがサイレントに詰まる可能性がある。暫定回避：バックグラウンドエージェントに渡すスクリプトは `--non-interactive` / `-y` など stdin 読み取りを発生させないフラグを明示的に付与する（例：`xcodebuild -quiet`、`brew install --yes`）。

## 💡 今日のおすすめ実践 Tip

**「v2.1.215 で `/verify` と `/code-review` が自動実行されなくなった — iOS 開発ワークフローへの組み込み方」**

v2.1.215 以降、Claude は実装後に自律的に `/verify` や `/code-review` を呼び出さなくなりました。これは「知らないうちに余計な検証が走る」ことがなくなるメリットがある一方、意図していた自動検証が行われなくなることを意味します。iOS 開発では特に「ビルドエラーが消えたタイミング」や「PR 前の最終確認」で明示的に呼ぶ習慣を付けておくと品質が維持できます。

**1. 手動呼び出しの基本フロー（XcodeBuildMCP と組み合わせ）**

```
# 実装が終わったら明示的に呼ぶ
/code-review

# ビルドエラー修正後に動作確認も含めて検証したい場合
/verify
```

**2. CLAUDE.md に「実装完了時のチェックリスト」として組み込む例**

v2.1.215 の変更を受け、自動実行に頼らずチェックを確実に実施するために CLAUDE.md に reminder を追記しておく方法が有効です：

```markdown
## 実装完了時のチェックリスト
すべての実装タスクが完了したら、必ず以下の順で実行してください：

1. `mcp__XcodeBuildMCP__BuildProject` でビルドが通ることを確認
2. `/code-review` でコードレビューを実施
3. ビルドエラー・レビュー指摘がなければ変更をコミット
```

**3. バックグラウンドセッション（claude agents）との組み合わせ注意点**

Issue #79175（バックグラウンド Bash の stdin ブロック）と v2.1.215 の修正を踏まえると、バックグラウンドエージェントでの iOS ビルドには次の点に注意が必要です：

| 状況 | 対策 |
|---|---|
| バックグラウンドで `xcodebuild` を呼ぶ | `-quiet` で不要なプロンプトを抑制 |
| `brew install` など対話コマンド | `--yes` / `--force` を付けて stdin 読み取りを回避 |
| `/verify` を自動で走らせたい | CLAUDE.md にチェックリストで明示指示（自動実行はされない） |

**4. Issue #79198（iOS アプリでのセッション消滅）への対策**

iPhone から Remote Control でビルドセッションを確認する運用をしている場合、iOS アプリの pull-to-refresh によってセッションが意図せずアーカイブされる可能性があります。以下の点を習慣にしてください：

- iPhone でセッション一覧を開く前に Mac のデスクトップアプリで接続状態を確認する
- セッションが消えていた場合は Desktop の「Archived」タブを確認してリストアする
- 重要なビルドセッションは Bark など別チャンネルの push 通知（`hooks: PostToolUse`）でバックアップとして状態通知を受け取る設定を用意しておく
